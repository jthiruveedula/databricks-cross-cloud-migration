# Data reconciliation

Data reconciliation proves that target data matches source data within agreed tolerances. It is the most critical validation activity before cutover.

## Reconciliation dimensions

| Dimension | Method | Tolerance |
|-----------|--------|-----------|
| Row count | `COUNT(*)` per table | 0% |
| Column-level checksum | Hash aggregate by column | 0% |
| File count | List files in storage | 0% |
| File size / checksum | Cloud-native checksums | 0% |
| Schema | Compare column names, types, nullability | Exact match |
| Partitions | Compare partition values and counts | 0% |
| Primary/unique keys | Verify uniqueness constraints | 0% |
| Delta table history | `DESCRIBE HISTORY` on target vs. source | Non-zero — see below |

## Row-count and checksum query

<CodeBlock client:visible language="sql" filename="reconcile.sql" code={`-- Reconcile row count
SELECT 'source' AS env, COUNT(*) AS cnt FROM source_catalog.schema.table
UNION ALL
SELECT 'target' AS env, COUNT(*) AS cnt FROM target_catalog.schema.table;

-- Column-level checksum
SELECT md5(concat_ws(',', cast(col1 AS string), cast(col2 AS string))) AS hash
FROM source_catalog.schema.table
ORDER BY hash;`} />

## Volume reconciliation

For non-tabular files:

<CodeBlock client:visible language="bash" code={`# AWS example: compare file counts and total size
aws s3 sync s3://source-bucket/path s3://target-bucket/path --dryrun

# GCP example
gsutil -m rsync -r gs://source-bucket/path gs://target-bucket/path -d -n`} />

## Delta table history reconciliation

<CodeBlock client:visible language="sql" filename="describe_history.sql" code={`DESCRIBE HISTORY source_catalog.schema.table;
DESCRIBE HISTORY target_catalog.schema.table;`} />

<Callout client:visible variant="warning" title="Zero history on target is a migration-method bug, not a data bug">
  If the target table's history is empty or starts only at the migration date, the table was very likely copied with `CREATE TABLE AS SELECT`, `INSERT OVERWRITE`, or a raw cloud-storage sync (`aws s3 sync`, `gsutil rsync`) instead of `CREATE TABLE ... DEEP CLONE`. Row counts and checksums will still match — this check exists specifically because those checks alone won't catch it. Time travel and audit queries against pre-migration versions will silently fail once source is decommissioned. Confirm which tables genuinely don't need history (see [FAQ](/troubleshooting/faq)) versus which lost it by accident.
</Callout>

## Handling mismatches

1. Identify the partition or file causing the mismatch.
2. Re-run the copy or sync for that subset.
3. Investigate if the mismatch is due to ongoing source changes during sync.
4. Document the resolution before signing off.

<Callout client:visible variant="warning" title="Do not ignore small mismatches">
  Even a single-row mismatch can indicate a bug in the migration logic or a permissions issue. Treat every mismatch as a severity-2 finding until explained.
</Callout>

## Validation

- Reconcile every migrated table and volume.
- Run reconciliation after final sync and again after cutover.
- Store reconciliation reports with timestamps.

## Rollback

If reconciliation fails beyond tolerance, halt cutover and re-sync. Do not proceed until all severity-1 findings are resolved.

## Automation opportunity

- Run reconciliation queries via a Databricks job and store results in a Delta table.
- Generate alerts when any table exceeds tolerance.
