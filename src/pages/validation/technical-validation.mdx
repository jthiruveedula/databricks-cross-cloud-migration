# Technical validation

Technical validation confirms that the target platform is functional, secure, and performant before users are cut over. It covers infrastructure, IAM, metastore, compute, pipelines, and integrations.

## Validation categories

| Category | Objective | Steps | Evidence |
|----------|-----------|-------|----------|
| Infra | Target environment is provisioned | Check workspace, region, SKU, networking | Console screenshot, Terraform plan |
| IAM | Users and principals can access | Log in, run privilege checks | Audit log, test results |
| Metastore | Metadata is complete and consistent | Compare catalog/schema/table counts | Reconciliation report |
| Compute | Clusters start and run | Start each cluster type, run smoke test | Cluster event logs |
| Pipelines | Jobs complete successfully | Run jobs and compare outputs | Job run history |
| SQL/Dashboards | Queries return expected results | Run dashboard queries | Query output diff |
| Security | Controls are enforced | Test denied access, verify encryption | Security scan report |
| Cost/Perf | Performance within SLA | Measure query latency and cost | Benchmark report |

## Infra validation script

<CodeBlock client:visible language="bash" code={`# Verify target workspace health
curl -n https://<target-workspace>.cloud.databricks.com/api/2.0/clusters/list \\
  -H "Authorization: Bearer $TOKEN" | jq '.clusters[] | {cluster_id, cluster_name, state}'

# Verify metastore attachment
databricks metastores list --output json | jq '.[] | {name, storage_root, region}'`} />

## IAM validation

<CodeBlock client:visible language="sql" code={`-- Verify current user and group membership
SELECT current_user();

-- Verify grants on a catalog
SHOW GRANT ON CATALOG <catalog_name>;`} />

## Validation sign-off

<Callout client:visible variant="decision" title="Sign-off gate">
  Require platform, security, and application owners to sign off on technical validation before cutover. Keep evidence in an immutable location.
</Callout>

## Rollback

If validation fails, fix the issue and re-run the affected test category. Do not proceed to cutover with open severity-1 or severity-2 findings.

## Automation opportunity

- Run validation tests as a CI/CD pipeline.
- Store results in a Delta table for audit and trending.
