# Terraform patterns

Use infrastructure as code to create repeatable target foundations. These patterns are cloud-agnostic in structure but use cloud-specific resources.

## Why this matters

Hand-writing target-cloud Terraform from scratch reproduces whatever the source workspace happens to look like today, drift and all. Generating a starting point from the source workspace's actual configuration — rather than only writing target resources by hand — catches settings teams forgot they had.

## Applicability

Use these patterns once the target cloud, region, and Unity Catalog topology are decided (see [Unity Catalog strategy](/governance/unity-catalog-strategy)). Not a substitute for the exporter step below when a source workspace configuration already exists and needs to be captured before it's decommissioned.

## Generating Terraform from source state

<Callout client:visible variant="tip" title="Export before you hand-write">
  Before writing target Terraform by hand, use the [Databricks Terraform exporter](https://registry.terraform.io/providers/databricks/databricks/latest/docs/guides/experimental-exporter) (`databricks exporter`, an experimental feature of the Databricks Terraform provider) to generate a baseline configuration from the *source* workspace. It captures clusters, jobs, notebooks, cluster policies, and (with scope flags) Unity Catalog objects as importable Terraform — giving you a real inventory of what exists today instead of relying on tribal knowledge of what should exist. Adapt the generated resources for the target cloud rather than starting from a blank module.
</Callout>

<CodeBlock client:visible language="bash" code={`# The exporter ships inside the databricks/databricks Terraform provider plugin, not as a
# separate CLI. Authenticate via DATABRICKS_HOST / DATABRICKS_TOKEN env vars (or let it
# prompt interactively), then run the provider binary from your Terraform plugin cache:
export DATABRICKS_HOST=<SOURCE_WORKSPACE_URL>
export DATABRICKS_TOKEN=<TOKEN>

terraform-provider-databricks exporter \\
  -skip-interactive \\
  -listing=compute,jobs,notebooks,policies \\
  -directory ./exported`} />

This is distinct from [UCX-assisted migration](/governance/ucx-strategy): the exporter captures workspace-level object configuration as IaC; UCX assesses and executes the Hive-metastore-to-Unity-Catalog upgrade. Large estates typically use both — UCX for the governance upgrade, the exporter for capturing workspace object configuration as a Terraform starting point.

## AWS target foundation

<CodeBlock client:visible language="hcl" filename="aws-target.tf" code={`terraform {
  required_providers {
    databricks = {
      source  = "databricks/databricks"
      version = ">= 1.36"
    }
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "databricks" {
  alias    = "mws"
  host     = "https://accounts.cloud.databricks.com"
  account_id = var.databricks_account_id
  username = var.databricks_account_user
  password = var.databricks_account_password
}

resource "databricks_mws_vpc_endpoint" "backend" {
  provider                   = databricks.mws
  account_id                 = var.databricks_account_id
  aws_vpc_endpoint_id        = aws_vpc_endpoint.backend.id
  vpc_endpoint_name          = "\${var.prefix}-backend"
  region                     = var.region
  depends_on                 = [aws_vpc_endpoint.backend]
}

resource "databricks_storage_credential" "uc" {
  provider     = databricks.mws
  name         = "\${var.prefix}-uc-storage"
  metastore_id = databricks_metastore.this.id
  aws_iam_role {
    role_arn = aws_iam_role.uc_storage.arn
  }
}

resource "databricks_metastore" "this" {
  provider      = databricks.mws
  name          = "\${var.prefix}-metastore"
  storage_root  = "s3://\${aws_s3_bucket.metastore.id}/metastore"
  region        = var.region
  force_destroy = true
}`} />

## Azure target foundation

<CodeBlock client:visible language="hcl" filename="azure-target.tf" code={`resource "azurerm_databricks_access_connector" "uc" {
  name                = "\${var.prefix}-uc-connector"
  resource_group_name = azurerm_resource_group.this.name
  location            = var.region
  identity {
    type = "SystemAssigned"
  }
}

resource "azurerm_role_assignment" "uc_storage" {
  scope                = azurerm_storage_account.uc.id
  role_definition_name = "Storage Blob Data Contributor"
  principal_id         = azurerm_databricks_access_connector.uc.identity[0].principal_id
}

resource "databricks_metastore" "this" {
  name         = "\${var.prefix}-metastore"
  storage_root = "abfss://\${azurerm_storage_container.uc.name}@\${azurerm_storage_account.uc.name}.dfs.core.windows.net/metastore"
  region       = var.region
}

resource "databricks_metastore_data_access" "uc" {
  metastore_id = databricks_metastore.this.id
  name         = "\${var.prefix}-uc-access"
  azure_managed_identity {
    access_connector_id = azurerm_databricks_access_connector.uc.id
  }
  is_default = true
}`} />

## GCP target foundation

<CodeBlock client:visible language="hcl" filename="gcp-target.tf" code={`resource "google_service_account" "uc" {
  account_id   = "\${var.prefix}-uc-sa"
  display_name = "Databricks UC Storage"
}

resource "google_storage_bucket_iam_member" "uc" {
  bucket = google_storage_bucket.uc.name
  role   = "roles/storage.objectAdmin"
  member = "serviceAccount:\${google_service_account.uc.email}"
}

resource "google_service_account_iam_member" "databricks_token_creator" {
  service_account_id = google_service_account.uc.name
  role               = "roles/iam.serviceAccountTokenCreator"
  member             = "serviceAccount:\${var.databricks_google_service_account}"
}

resource "databricks_metastore" "this" {
  name         = "\${var.prefix}-metastore"
  storage_root = "gs://\${google_storage_bucket.uc.name}/metastore"
  region       = var.region
}`} />

<Callout client:visible variant="tip" title="Modularize">
  Keep cloud-specific modules separate and use a common variables interface. This makes multi-cloud migrations easier to maintain and review.
</Callout>

## Validation

- Run `terraform plan` against the target account and confirm zero unexpected diffs before every apply — especially after re-running the exporter, since re-imports can introduce drift in provider-managed fields.
- Verify the metastore, storage credential, and access connector resources actually resolve (test a `CREATE TABLE` and `GRANT` in the target) before treating the foundation as complete.

## Rollback

Target foundation resources created via Terraform should be destroyed via `terraform destroy` scoped to the specific module, not manually in the cloud console — manual deletion causes state drift that blocks future applies. Keep `terraform.tfstate` for the target foundation in remote state (not local disk) so rollback is possible even if the machine that ran `apply` is gone.

## Automation opportunity

- Run `terraform plan` in CI on every pull request touching the foundation modules, and require a human-reviewed `plan` output before `apply`.
- Store per-cloud variable files (`aws.tfvars`, `azure.tfvars`, `gcp.tfvars`) in version control so the same module set can be re-applied for future migrations without rediscovering variable names.

## Evidence to capture

- The `terraform plan` and `apply` output for the target foundation, archived alongside the migration's decision register.
- The exporter's output diff (source configuration vs. what was actually carried into the target module) so reviewers can see what was intentionally left out.

## Cloud-specific caveats

The `databricks` Terraform provider's resource names and required arguments differ meaningfully across clouds even where the concept is shared — `databricks_storage_credential` and `databricks_metastore_data_access` take an `aws_iam_role { role_arn = ... }` block on AWS, an `azure_managed_identity { access_connector_id = ... }` block on Azure (the Azure Databricks Access Connector, not a raw managed identity, is the actual UC storage-credential object), and a Databricks-managed `databricks_gcp_service_account` block on GCP — the three are not interchangeable, and provider schema for these resources has changed across releases. Always check the exact schema for your pinned provider version rather than assuming argument names transfer from the AWS example to Azure/GCP.
