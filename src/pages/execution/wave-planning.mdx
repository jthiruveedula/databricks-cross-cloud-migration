# Wave planning

Divide the migration into waves that move related workloads together. Waves reduce blast radius, let teams learn, and make rollback simpler.

## Wave design principles

- Group by **dependency chain**, not by asset type. A wave is "the table, the pipeline that populates it, and every dashboard/job that reads it" — never just "all the dashboards" or "all the pipelines" as separate waves. See [dependency mapping](/discovery/dependency-mapping) for how to derive these clusters instead of guessing.
- Group by business domain within that constraint, not by cloud primitive.
- Start with low-risk, non-production workloads.
- Reserve the final wave for the most critical or tightly coupled systems.

<Callout client:visible variant="warning" title="A dashboard has no data until its pipeline has migrated too">
  If Wave 1 moves "the dashboards" and Wave 2 moves "the batch pipelines" as separate asset-type waves, Wave 1's dashboards have nothing current to query -- the job that refreshes their source tables is still running against source, not target. Every wave below bundles a reporting/domain's tables, the pipeline(s) that populate them, *and* the dashboards that read them together; "Analytics and BI" names the wave's business domain, not a list of only-dashboards.
</Callout>

## Sample wave structure

  {
    title: 'Wave 3: Streaming and ML domains',
    content: 'Real-time pipelines, model serving, and feature stores, each with their own upstream data and downstream consumers migrated together. Highest operational risk.'

  }
]} />

## Wave scoring matrix

| Criterion | Weight | Notes |
|-----------|--------|-------|
| Business criticality | High | Critical workloads go late or use blue-green |
| Data size | Medium | Large data drives longer sync windows |
| Dependency count | High | High-dependency workloads need careful sequencing |
| Risk tolerance | High | Low-tolerance workloads need more validation |
| Owner readiness | Medium | Teams must be available during cutover |

## Wave plan template

<CodeBlock client:visible language="yaml" filename="wave-plan.yaml" code={`waves:
  - id: wave-0
    name: Pilot
    workspaces:
      - workspace_id: <WS_ID>
        target_workspace_id: <TWS_ID>
    catalogs:
      - pilot_analytics
    criteria:
      entry:
        - discovery complete
        - target foundation ready
      exit:
        - smoke tests pass
        - business sign-off received
    rollback:
      - re-enable source jobs
      - redirect users to source workspace
}`} />

## Validation

- Each wave must have clear entry and exit criteria.
- Dependencies across waves must be documented and signed off.
- A wave should not start until the previous wave exits successfully.

## Rollback

Rollback is performed per wave. If a wave fails, revert the workloads in that wave and keep earlier waves in the target.

## Automation opportunity

- Generate wave manifests from dependency graphs.
- Use orchestration tools to trigger validation and cutover steps per wave.
