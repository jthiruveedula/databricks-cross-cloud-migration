# UCX-assisted migration

[UCX](https://github.com/databrickslabs/ucx) (Databricks Labs' Unity Catalog eXpress) is the primary open-source tool for assessing and executing **workspace-local Hive metastore → Unity Catalog** upgrades. It is not a cross-cloud migration tool by itself — it has no concept of a target cloud, target region, or target account — but it is the highest-leverage accelerator for the governance-upgrade work that almost every cross-cloud migration must also do. Treat UCX as one stage in a broader migration factory, not the whole migration.

## Why this matters

Most enterprise Databricks estates still have a meaningful share of workspace-local Hive metastore tables, DBFS mounts, and workspace-local groups. Combining "upgrade to Unity Catalog" with "move to a different cloud" in one uncoordinated effort is how migrations blow their timelines — UCX exists to make the first part mechanical and auditable so the team's attention stays on the genuinely cross-cloud work (identity, storage, networking).

## Applicability

- **Use UCX when:** the source workspace still has `hive_metastore`-catalog tables, workspace-local groups, or DBFS-mount-heavy notebooks/jobs that need to be inventoried and upgraded to Unity Catalog — regardless of whether the target is the same cloud or a different one.
- **Don't rely on UCX alone when:** the migration is workspace-to-workspace across clouds. UCX upgrades objects *in place, within a source workspace's account*. It does not create target-cloud storage credentials, does not provision a target metastore in a different account, and does not move data across clouds. Those steps still come from [Unity Catalog strategy](/governance/unity-catalog-strategy), [metastore migration](/governance/metastore-migration), and the per-cloud [Terraform patterns](/templates/terraform-patterns).
- **Skip UCX when:** the source is already fully on Unity Catalog with account-level groups and no legacy Hive/DBFS surface — go directly to metastore migration.

## Inputs required

- Account admin (or workspace admin with UCX-required scopes) access to the source workspace.
- A dedicated "assessment" workspace to install UCX into — UCX writes its findings as Delta tables and dashboards there, not into the workspace being assessed.
- An inventory of workspace-local groups already gathered from [workspace inventory](/discovery/workspace-inventory).

## Recommended sequence

<Callout client:visible variant="decision" title="UCX has three broad workflows, run in this order">
  Assessment always comes first and is non-destructive. Group migration and table migration are separate, gated workflows — do not run them back-to-back without reviewing the assessment output between them.
</Callout>

### 1. Assessment workflow (read-only)

Scans the workspace for Hive metastore tables, DBFS mount usage, workspace-local groups, cluster configurations incompatible with Unity Catalog access modes, and job/notebook code referencing legacy paths. Produces a readiness report as Delta tables plus a dashboard. Nothing in the source workspace is modified.

<CodeBlock client:visible language="bash" code={`# Install UCX into the workspace (prompts for the assessment job's warehouse/cluster config)
databricks labs install ucx

# Trigger the assessment workflow and block until it completes
databricks labs ucx ensure-assessment-run

# Check status of all UCX-deployed workflows (assessment, migrate-groups, etc.)
databricks labs ucx workflows

# Export the readiness report (CSV/zip by default, or --export-format excel)
databricks labs ucx export-assessment`} />

<p className="text-xs text-[var(--ink-subtle)]">Verified against the current UCX CLI reference (databrickslabs.github.io/ucx) — confirm against the version you install, since UCX's command surface has changed across releases and this site cannot track it live.</p>

### 2. Group migration workflow

Converts workspace-local groups to account-level groups and re-points permissions (cluster policies, job ACLs, notebook ACLs) at the new account-level group. This is a prerequisite for Unity Catalog grants to behave correctly, whether or not a cloud change is involved. `migrate-groups` runs as a deployed Databricks Job, not a direct CLI subcommand — trigger and monitor it through the `workflows` status table and `repair-run`, not a "run workflow" flag.

<CodeBlock client:visible language="bash" code={`# Create the account-level groups that mirror workspace-local ones
databricks labs ucx create-account-groups

# The migrate-groups workflow is a Databricks Job UCX deployed at install time —
# start/monitor it from the Databricks Jobs UI or the Databricks CLI jobs API,
# then confirm completion and re-run failures from here:
databricks labs ucx workflows
databricks labs ucx repair-run --step migrate-groups

# Confirm no workspace-local groups remain assigned to clusters/jobs/notebooks
databricks labs ucx validate-groups-membership`} />

### 3. Table migration workflow

Upgrades `hive_metastore`-catalog tables to Unity Catalog catalogs/schemas using the strategy indicated by the assessment (in-place `SYNC`, `CTAS`, or `DEEP CLONE`, depending on table format and location).

<CodeBlock client:visible language="bash" code={`# Build and edit the source-to-target catalog/schema mapping
databricks labs ucx create-table-mapping

# Create the target catalogs and schemas from that mapping
databricks labs ucx create-catalogs-schemas

# Run the table migration itself
databricks labs ucx migrate-tables`} />

## Readiness reports

UCX's assessment dashboard is the artifact to socialize with stakeholders before committing to a wave plan — it typically surfaces, per table: format (Delta/Parquet/other), location type (DBFS mount / cloud-native path / workspace-local path), and a recommended upgrade strategy. Export this into the [risk register](/templates/risk-register) and [wave scoring](/execution/wave-planning) inputs rather than re-deriving readiness by hand.

## Limits of UCX

<Callout client:visible variant="warning" title="UCX does not cross clouds">
  UCX operates entirely within the source workspace's existing cloud account. It has no mechanism to create a target-cloud metastore, target storage credentials, or replicate data to a different cloud's object store. For a genuinely cross-cloud migration, UCX handles the Hive→UC upgrade *before or in parallel with* the cross-cloud data and identity work — it does not replace it.
</Callout>

- Does not migrate ML artifacts, MLflow experiments, or model registry entries — see [MLflow](/ml/mlflow) and [model registry](/ml/model-registry).
- Does not migrate dashboards, alerts, or SQL warehouse configuration — see the [Analytics](/analytics/sql-queries) section.
- Grant migration during table upgrade assumes the mapped account-level group already exists; it will not invent group mappings for you.
- Assessment quality depends on job/notebook code being statically analyzable — dynamically constructed table references (string-built paths, `getArgument`-driven table names) are frequently missed and need manual review.

## How UCX fits into a broader factory model

Treat UCX as the tooling for one lane of the migration factory (see [wave planning](/execution/wave-planning)):

| Factory stage | UCX's role |
|---|---|
| Intake / assessment | UCX assessment workflow feeds the per-asset readiness score |
| Governance upgrade | UCX group + table migration workflows execute the Hive→UC conversion |
| Cross-cloud foundation | Out of scope for UCX — handled by Terraform patterns and target metastore provisioning |
| Data replication | Out of scope for UCX — handled by Deep Clone / CTAS per [metastore migration](/governance/metastore-migration) |
| Validation | UCX's post-migration checks feed [technical validation](/validation/technical-validation), not a replacement for it |

## Validation

- Diff UCX's assessment table/group counts against the [workspace inventory](/discovery/workspace-inventory) and [asset inventory](/discovery/asset-inventory) exports — a mismatch means UCX missed dynamically referenced objects.
- Re-run the assessment workflow after group migration and confirm zero remaining workspace-local-group references before starting table migration.
- Spot-check a sample of UCX-upgraded tables for correct owner and grant assignment, not just successful upgrade status.

## Rollback

UCX's assessment workflow is non-destructive and needs no rollback. Group migration and table migration are not fully reversible in place — keep the source Hive metastore tables and workspace-local groups untouched (do not delete) until the corresponding Unity Catalog objects are validated, so reverting means re-pointing consumers back to the original objects rather than restoring deleted state.

## Automation opportunity

- Run the assessment workflow on a schedule during discovery so the readiness report stays current as the estate changes.
- Feed UCX's per-table recommended strategy directly into the wave-scoring inputs instead of re-classifying tables manually.
- Version-control the group mapping UCX generates so grant replay ([grants and roles](/governance/grants-and-roles)) is reproducible.

## Evidence to capture

- UCX assessment dashboard export (per-table readiness, per-group mapping) — timestamped, kept as the pre-migration baseline.
- Group migration workflow run log and before/after ACL diff for a sample of clusters and jobs.
- Table migration workflow run log, per-table upgrade strategy applied, and any tables flagged as manual-review-required.

## Cloud-specific caveats

UCX itself is cloud-agnostic — it operates against the Databricks workspace/account APIs, not cloud-native APIs directly. The caveats are downstream of UCX's output: a table UCX marks "ready to upgrade to managed Delta" still needs a target-cloud storage credential and external location created (per-cloud specifics in [external locations and volumes](/governance/external-locations-and-volumes)) before the corresponding cross-cloud table migration step can run.
