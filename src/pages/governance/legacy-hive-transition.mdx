# Legacy Hive transition

A migration is the ideal time to retire the legacy Hive metastore and its DBFS-mount dependencies for good — carrying them into the target cloud only defers the same cutover pain to a future migration. This page covers table-level upgrade paths; for the file-level mount retirement mechanics (inventorying `dbutils.fs.mounts()`, rewriting paths, unmounting), see [external locations and volumes](/governance/external-locations-and-volumes).

## Why this matters

Workspace-local Hive metastores and DBFS mounts are the two most common sources of "it worked in the old workspace but breaks in the new one" incidents during cross-cloud cutover: both are implicitly tied to a single workspace's identity and storage configuration, and neither concept exists in a cross-cloud target the same way. Deferring this work does not avoid it — it just moves the failure from planning time to cutover night.

## Why retire Hive now?

- Workspace-local metastores do not share governance across workspaces — every workspace has its own, so nothing here is portable by definition.
- DBFS-backed managed tables are tied to a workspace and are not portable; the data and the workspace-local metastore pointer to it must both move, which is exactly what Unity Catalog is designed to avoid needing again.
- Legacy runtimes and access modes will not work with Unity Catalog compute.

## Applicability

Applies to any source workspace still running the legacy (workspace-local) Hive metastore, or one that has UC enabled but still has un-upgraded Hive-backed tables and DBFS mounts alongside it. Does not apply if the source workspace is already fully on Unity Catalog with no remaining `hive_metastore` catalog objects — skip straight to [metastore migration](/governance/metastore-migration).

## Upgrade paths

| Source table type | Target type | Method |
|-------------------|-------------|--------|
| Hive managed (DBFS) | UC managed | `CREATE TABLE target DEEP CLONE hive_table` |
| Hive external | UC external | `CREATE EXTERNAL TABLE target LOCATION '...'` |
| Hive view | UC view | Recreate view after base tables |

## Coexistence and federation

If a full upgrade is not possible before cutover:

- Use Hive metastore federation to query Hive tables from UC-enabled clusters.
- Gradually migrate tables wave by wave.
- Avoid creating new Hive assets in the source during the migration.

## Validation

- Run UCX assessment and resolve critical findings before migration.
- Confirm all UC-bound compute uses a supported access mode and DBR 13.3 LTS or above.
- Validate that upgraded tables are queryable with Unity Catalog clusters.

## Rollback

Keep the original Hive tables until target tables are validated. Do not drop Hive assets until hypercare is complete.

## Automation opportunity

- Use UCX-generated notebooks for bulk upgrades.
- Schedule upgrade jobs per schema and validate with reconciliation queries.
