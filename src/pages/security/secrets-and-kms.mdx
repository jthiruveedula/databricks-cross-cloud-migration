# Secrets and KMS

Secrets and encryption keys are never portable across clouds. You must recreate secret scopes, rekey encryption keys, and rotate credentials as part of migration.

## Secret scope mapping

| Source | Target | Action |
|--------|--------|--------|
| Azure Key Vault-backed scope | AWS Secrets Manager-backed scope | Create new scope and copy secrets |
| Databricks-backed scope | Databricks-backed scope | Export and import via Secrets API |
| AWS Secrets Manager | GCP Secret Manager | Recreate scope and secrets |

<Callout variant="warning" title="Rotate all credentials">
  Treat migration as a credential rotation event. Do not copy secrets unchanged if they protect production systems.
</Callout>

## Migrating secrets

<CodeBlock client:visible language="bash" code={`# List source secrets
databricks secrets list --scope production --output json

# Create target scope backed by target cloud secret store
databricks secrets create-scope --scope production --scope-backend-type AZURE_KEYVAULT --resource-id <target-vault-id> --dns-name <target-vault-uri>

# Put secret (repeat for each)
databricks secrets put --scope production --key api-key --string-value <new-value>}`} />

## Customer-managed keys

| Source | Target |
|--------|--------|
| Azure Key Vault CMK | AWS KMS customer-managed key |
| AWS KMS CMK | Azure Key Vault key or GCP Cloud KMS key |
| GCP Cloud KMS CMEK | AWS KMS or Azure Key Vault key |

Plan key rotation and data re-encryption:

- If data is copied to target storage, encrypt it with the target CMK at write time.
- For blue-green, rotate target keys after cutover and disable old key versions per policy.

## Personal access token retirement

<Callout variant="tip" title="Use OAuth and service principals">
  Replace personal access tokens with OAuth service principals and managed identities. PATs tied to individual users do not survive identity changes and are harder to audit.
</Callout>

## Validation

- Resolve every secret reference in notebooks, jobs, and init scripts in a target scope.
- Test key access and rotation procedures in a non-production target workspace.
- Confirm cloud KMS policies allow the target compute identity to decrypt data.

## Rollback

If a secret is missing or wrong, the affected job fails. Maintain a pre-cutover secret manifest and a fast re-rotation runbook.

## Automation opportunity

- Export secrets metadata (not values) and generate target secret creation scripts.
- Use infrastructure as code to define KMS keys and secret scopes.
