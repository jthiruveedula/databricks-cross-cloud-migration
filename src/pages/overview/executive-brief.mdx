# Executive brief

This runbook is built for the team doing a Databricks migration **from one cloud to another** — AWS ↔ Azure ↔ GCP. If your goal is an in-place Unity Catalog upgrade or a same-cloud region move, most of this still applies, but start at [remote governance vs. migration](/overview/remote-governance-and-when-not-to-migrate) before committing to a physical move.

## The one-paragraph case

A cross-cloud migration is rarely about the data. It is about **identity, storage layout, networking, and governance** landing in a target account that your security and platform teams actually own. The data itself moves through Delta and object storage with well-trodden patterns; the schedule slips on everything around the data. Treat the move as a platform-modernization program with a data-migration workstream attached, not the reverse.

## The phase model

Every cross-cloud migration in this runbook flows through the same seven phases. Duration and cost scale with workspace count, user count, and how much legacy Hive metastore you are still carrying.

| Phase | What it produces | Typical share of timeline |
| --- | --- | --- |
| 1. Discovery | Asset + dependency inventory, risk register | 10–15% |
| 2. Foundation | Landing zone, identity federation, network | 15–20% |
| 3. Data & compute migration | Catalog, storage, clusters moved | 30–40% |
| 4. Pipeline migration | Jobs, DLT/Lakeflow, CI/CD | 10–15% |
| 5. Validation | Technical, data, security, business sign-off | 10–15% |
| 6. Cutover | Wave execution, rollback plan | 5–10% |
| 7. Hypercare | Monitoring, handover, decommission | 5–10% |

## The decision gate (do this first)

Before any wave plan, answer one question: **does the business driver require moving data at all?** Unity Catalog federation lets you govern and query data in place, across clouds, without a physical migration. A governance-only path is often measured in weeks; a physical migration in quarters.

- If the driver is a **single pane of governance**, compliance boundary, or read access for a new team → favor remote governance.
- If the driver is **egress cost, latency to a new region, a cloud mandate, or a hard sovereignty boundary** → physical migration is required.

## Top three risks that blow timelines

1. **Uninventoried dependencies.** A notebook that nobody owns calls a table in another workspace that depends on a DBFS mount. Discovery is cheap insurance.
2. **Identity done late.** Mapping principals, groups, and service principals across clouds is the longest pole — start it in Foundation, not Cutover.
3. **Legacy Hive metastore debt.** The more workspace-local tables and mounts you have, the more UCX-assisted upgrade work sits inside the migration.
