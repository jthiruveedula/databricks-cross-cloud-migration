# Decision framework

Use this framework to move from a business goal to a concrete migration approach. Each decision constrains the next, so do not skip steps.

## Step 1: Anchor on the business driver

| Driver | Implication |
|--------|-------------|
| Cloud standardization | Full cross-cloud migration; identity reset required. |
| M&A | Likely multi-tenant consolidation; data sovereignty may dictate region. |
| Regulatory isolation | Region and network boundaries dominate; blue-green recommended. |
| Cost optimization | Same-cloud move or storage-tier change; egress math is critical. |
| Regional expansion | Same-cloud new region or cross-cloud; data copy is unavoidable. |
| Security posture | Identity, networking, and encryption are first-class workstreams. |

## Step 2: Determine data gravity

Estimate data size, mutation rate, and downstream dependencies. Large or rapidly changing data sets favor incremental sync and blue-green cutover. Static or small data sets can use lift-and-shift within a single maintenance window.

## Step 3: Choose the migration archetype

See [Migration archetypes](migration-archetypes). The most common enterprise choice is **workspace + metastore** with a phased wave plan.

## Step 4: Decide on Unity Catalog strategy

<Callout variant="tip" title="UC-first">
  If the source is not on Unity Catalog, combine the migration with UC adoption. Do not migrate into a legacy Hive metastore on the target.
</Callout>

Choices:

- **Federate first:** Register source storage as a foreign catalog, then upgrade tables in place.
- **Clone and upgrade:** Deep clone Delta tables into a target-managed catalog, preserving history.
- **Rebuild:** Recreate schemas and load data with CTAS / COPY INTO for non-Delta sources.

## Step 5: Define cutover model

| Model | Downtime | Risk | Cost | Use case |
|-------|----------|------|------|----------|
| Big-bang | Hours to days | High | Low | Small estate, tolerant users |
| Incremental | Minimal | Medium | Medium | Large tables, daily sync |
| Blue-green | Near-zero | Low | High | Mission-critical workloads |

## Validation

- Capture each decision in a decision register with owner, date, and assumptions.
- Revisit the framework after discovery if the estate is larger or more complex than expected.

## Rollback

The framework itself has no rollback, but every decision should have a contingency. For example, if blue-green cost is rejected, have an incremental fallback ready.

## Automation opportunity

Store the decision register and wave plan in a version-controlled YAML file. Example:

  source:
    cloud: azure

    account_id: <TARGET_ACCOUNT_ID>
    region: us-west-2

    - id: wave-1

      criteria: non-production, no PII, <1 TB
    - id: wave-2
      name: Analytics workloads
      criteria: SQL dashboards and batch jobs
    - id: wave-3
      name: Critical pipelines
      criteria: requires dual-run sign-off
}`} />
