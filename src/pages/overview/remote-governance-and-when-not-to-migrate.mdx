# Remote governance vs. migration: when not to move data

Physically migrating data across clouds is expensive, slow, and carries egress cost and risk that a governance-only approach can avoid entirely. Before committing to the wave plan in [decision framework](/overview/decision-framework), ask whether the business driver actually requires moving data at all — Unity Catalog's federation capabilities let you govern and query data in place, across clouds, without a physical migration. This page is the decision gate that belongs before, not after, archetype selection.

## Why this matters

Teams that jump straight to "we're migrating to cloud X" without asking this question routinely discover mid-migration that the actual business requirement — a single pane of governance, a compliance boundary, or query access for a new team — could have been met in weeks with federation instead of the quarters a physical migration takes. Getting this decision wrong is expensive in both directions: over-migrating wastes egress and cutover risk; under-migrating (governing remotely when physical migration was actually required) leaves latency, cost, or sovereignty problems unsolved.

## Applicability

Evaluate this decision whenever the driver is one of: a new consumer needs read access to data that lives in another cloud, a merger/acquisition needs a unified catalog across two estates on different clouds, a compliance program needs centralized audit/lineage without disturbing data residency, or leadership has mandated "get everything on one cloud" without a clear technical reason. It does not apply once a genuine technical driver is present — sustained low-latency compute co-location, egress cost of repeated cross-cloud reads exceeding one-time migration cost, or a hard data-residency requirement that federation cannot satisfy.

## The three options

| Option | What it is | Best when |
|---|---|---|
| **Migrate** | Physically move data, compute, and identity to the target cloud; source is decommissioned | Compute needs to run where the data lives (cost, latency), or the source platform/account is being retired entirely |
| **Govern remotely (federate)** | Register the source cloud's storage as a Unity Catalog foreign catalog / external location from a target-cloud metastore; query in place, no data copy | A new team or workload needs governed access to data that has no compute co-location requirement, or a full migration isn't yet funded/scheduled |
| **Hybrid** | Federate first for immediate governed access, migrate physically later on its own timeline | M&A consolidation, or any case where governance urgency and data-movement urgency are on different clocks |

## What remote governance looks like in practice

- **Unity Catalog external locations / storage credentials** pointing at the *other* cloud's storage, registered from a metastore in the cloud where governance and query compute live.
- **Foreign catalogs / Lakehouse Federation** connections for querying data that stays in its original system or cloud without copying it into Delta first.
- **Delta Sharing** for read-only cross-cloud (or cross-organization) access patterns where the consumer doesn't need write access or low-latency joins against the shared data.
- **Read-only cross-cloud access** for a defined, narrow consumer set — e.g., a new analytics team on Azure needs read access to AWS-resident tables while the rest of the estate stays put.

## "When not to migrate" — related decision branches

Even when a cross-cloud driver is real, physical migration is sometimes still the wrong next step:

- **Governance-only is enough:** the requirement is audit/lineage/access-policy consistency, not data movement — see the federation options above.
- **Same-cloud redesign is better:** the actual pain is workspace sprawl or poor account structure within the current cloud, not the cloud itself — a same-cloud landing-zone move (see [migration archetypes](/overview/migration-archetypes)) resolves it without cross-cloud risk.
- **Workspace cleanup should happen first:** migrating unused clusters, orphaned jobs, and stale permissions cross-cloud just relocates technical debt — clean up in place first (see [asset inventory](/discovery/asset-inventory) and [risk assessment](/discovery/risk-assessment)).
- **Unity Catalog adoption should complete before the cloud move:** if the source is still on a workspace-local Hive metastore, finish that upgrade in place (see [legacy Hive transition](/governance/legacy-hive-transition) and [UCX-assisted migration](/governance/ucx-strategy)) before adding a cross-cloud variable to the same effort.
- **Platform standardization should happen before workload migration:** if cluster policies, CI/CD promotion, and naming conventions aren't standardized in the source, migrating inconsistent workloads cross-cloud will bake the inconsistency into the target. Standardize, then migrate.

## Recommended sequence

1. Identify the actual business driver (see [decision framework](/overview/decision-framework) Step 1) before assuming migration is the answer.
2. Run the decision criteria checklist above.
3. If federation is sufficient, design the foreign-catalog / external-location topology and stop — no wave plan needed.
4. If hybrid, federate immediately for the urgent consumer, then run the full migration process on its own timeline and wave plan.
5. If migration is the answer, proceed to [migration archetypes](/overview/migration-archetypes) and the full decision framework.

## Validation

- Confirm the chosen option was documented with the specific driver and egress/cost math that justified it, not just team preference.
- For federation: validate query performance and cost against the read pattern's actual latency tolerance before declaring it a permanent solution.
- For hybrid: confirm the later migration timeline is tracked as a real commitment, not an indefinitely deferred "eventually."

## Rollback

Federation has a low-risk rollback: drop the foreign catalog / external location registration; no data was moved, so nothing to restore. A hybrid decision that later reveals migration is actually required simply proceeds to the standard migration rollback boundaries in [rollback](/execution/rollback).

## Automation opportunity

- Build the egress-cost comparison (one-time migration transfer vs. cumulative federated-query egress) as a reusable calculator fed by actual query volume from `system.access` tables, rather than a one-off spreadsheet per decision.
- Track federation-vs-migration decisions in the same decision register recommended in the [decision framework](/overview/decision-framework).

## Evidence to capture

- The documented driver, the egress/cost comparison, and the chosen option (migrate / govern remotely / hybrid), with sign-off.
- For federation: query latency and cost samples from the first weeks of production use, to confirm the assumption still holds.

## Cloud-specific caveats

Lakehouse Federation and Delta Sharing connection types and supported source systems vary by Databricks release and are not identical across AWS, Azure, and GCP deployments — confirm current support for your specific source/target pairing before committing to a federation-only strategy, especially for less common source systems (non-Databricks warehouses, on-prem sources).
