# DLT and Lakeflow

**Lakeflow Declarative Pipelines** is Databricks' current name for what was previously branded Delta Live Tables (DLT) — the underlying pipeline engine and `CREATE FLOW`/`@dlt.table`-style declarative model are the same lineage, not a rewrite, so existing DLT pipelines and terminology in your source workspace are not obsolete. This page uses "pipeline" generically for both. They must be recreated in the target workspace because pipelines are bound to the workspace, storage, and metastore.

## Why this matters

A declarative pipeline's checkpoint state and streaming-source configuration are two of the most commonly under-scoped parts of a pipeline migration — teams correctly plan for recreating the pipeline definition and target catalog, then discover during cutover that checkpoints don't transfer as cleanly as a table does, and that switching the underlying streaming source (not just its cloud location) invalidates checkpoint state entirely.

## Applicability

Applies to any Lakeflow/DLT pipeline being moved to a new workspace, whether or not the cloud changes. The streaming-source-swap caveat below applies specifically when the migration also changes the cloud-native event/streaming source a pipeline reads from (e.g., moving off Kinesis onto Event Hubs or Pub/Sub) — same-cloud pipeline moves with an unchanged source have a smaller version of this problem.

## Migration scope

- Pipeline definitions (JSON or Python notebooks).
- Target schemas, catalogs, and storage locations.
- Compute configuration and photon settings.
- Data quality expectations and notifications.
- Permissions and ownership.

## Streaming checkpoint migration

For pipelines with streaming sources, checkpoint handling is not a copy-the-files problem — treat it as its own decision, separate from copying the pipeline definition and target tables:

- **Checkpoint location portability:** a Structured Streaming checkpoint stores offsets/commit logs keyed to the exact storage path and, for some sources, absolute references back to the source system. Copying the checkpoint's files to target storage does not make it "the same checkpoint" if the path or source identity changed underneath it — validate a test restart before trusting a copied checkpoint in production.
- **Changing the streaming source connector invalidates checkpoints, regardless of copy strategy.** If the migration also swaps the underlying event source — AWS Kinesis → Azure Event Hubs → GCP Pub/Sub, for example — the new connector has no concept of the old source's offsets. This is not a path-rewrite problem to solve by copying checkpoint files; it requires a deliberate fresh-start decision (see below) because the old checkpoint literally cannot resume against a different source system.
- **Watermark and exactly-once semantics:** if the pipeline relies on watermarking or exactly-once sink semantics, confirm the target sink/source pairing preserves those guarantees — don't assume a "should behave the same" swap is exactly equivalent, particularly across differing at-least-once vs. exactly-once delivery models between cloud-native queue/stream services.
- Starting fresh may reprocess historical data; ensure downstream consumers and sinks are idempotent before choosing this path.

<Callout client:visible variant="warning" title="Checkpoints are not portable across a source-connector change">
  If the migration changes the streaming source (not just its cloud location), plan for a fresh-start checkpoint as the default assumption, not an edge case — the old checkpoint's offsets have no meaning against a different source connector. Validate this decision explicitly per pipeline rather than assuming a uniform strategy across all streaming pipelines in the wave.
</Callout>

## Validation

- Run the target pipeline and compare output row counts and quality metrics.
- Verify expectations and alerts.
- Measure latency and cost against source.

## Rollback

If the target pipeline produces incorrect output, stop it, fix the definition, and re-run. Keep the source pipeline available until validated.

## Automation opportunity

- Store pipeline definitions as code and deploy via CI/CD.
- Parameterize catalog and storage so the same code targets source or target.
