# Azure → GCP

Entra ID federation, ADLS Gen2 paths, and VNet Private Link have no direct GCP equivalent — identity, storage, and networking are rebuilt against Google Cloud IAM, Cloud Storage, and VPC with Private Service Connect. See the [cross-cloud equivalency matrix](/cloud-mappings/cross-cloud-equivalency-matrix) for the full reference; this page sequences the Azure → GCP-specific work.

## Sequencing

1. **Foundation first.** Provision the GCP project(s), Databricks account-level Unity Catalog metastore, VPC, and service accounts before touching data.
2. **Identity before data.** Federate Google Cloud Identity / Workspace SSO so users and groups resolve identically before any table is registered.
3. **Storage and paths.** Provision GCS buckets 1:1 with ADLS Gen2 containers, replicate data, then rewrite paths.
4. **Compute and jobs last.** Rebuild cluster policies, init scripts, and Workflows/DLT pipelines once storage and identity are validated.

## Identity federation

| Azure construct | GCP equivalent | Notes |
|---|---|---|
| Entra ID tenant | Google Cloud Identity / Workspace | Group names and IDs rarely match; rebuild the mapping explicitly |
| Managed identity (cluster) | Service account attached to cluster | Grant at the project/dataset level, not per-notebook |
| Service principal (automation) | Service account with key or impersonation | Prefer workload identity federation over long-lived keys |
| Entra ID SCIM provisioning | Google Workspace / Cloud Identity SCIM | Audit for orphaned groups after cutover |

<Callout client:visible variant="warning" title="Service account keys are a step backward">
  Don't replicate Entra ID service principal secrets as long-lived GCP service account JSON keys. Use workload identity federation or short-lived tokens — carrying over a "secret in a vault" pattern from Azure Key Vault without adopting GCP's federation model reintroduces a credential-leak risk you were trying to close.
</Callout>

Register the replicated data as a Unity Catalog **external location** in the target metastore, backed by a **Databricks-managed GCP service account** storage credential (`databricks_gcp_service_account` on `databricks_storage_credential`) — not a manually created service-account key. See [external locations and volumes](/governance/external-locations-and-volumes) for the general pattern and [Terraform patterns](/templates/terraform-patterns) for provisioning it as code.

<CodeBlock client:visible language="python" filename="path_rewrite.py" code={`MAPPING = {
    "abfss://raw@prodadls.dfs.core.windows.net": "gs://prod-raw-us-central1",
    "abfss://curated@prodadls.dfs.core.windows.net": "gs://prod-curated-us-central1",
}

def rewrite_path(path: str) -> str:
    for azure_prefix, gcs_prefix in MAPPING.items():
        if path.startswith(azure_prefix):
            return path.replace(azure_prefix, gcs_prefix, 1)
    raise ValueError(f"No mapping for path: {path}")
`} />

## Networking

- Replace **VNet + Private Link** with **VPC + Private Service Connect (PSC)**; the Databricks control-plane connection is re-established per workspace.
- Azure NAT Gateway egress rules do not transfer — recreate them against Cloud NAT.
- If the source used secure cluster connectivity (no public IPs), replicate that with **Private Google Access** on the GCP subnet rather than defaulting to public IPs.

## Compute and jobs

- Rebuild cluster policies against GCP machine types — Azure VM SKUs have no direct GCP match; size by vCPU/memory, not by name.
- Re-point service accounts (not managed identities) on every cluster policy and job cluster spec.
- Re-create Workflows/DLT pipelines against the new Unity Catalog catalogs — job IDs do not migrate; re-establish ownership and alerting.

## Validation

- Reconcile row counts and checksums between ADLS source tables and their GCS-backed Unity Catalog counterparts before cutover.
- Pilot the full IAM mapping with a read-only workload first — see [IAM mapping](/security/iam-mapping).
- Confirm CI/CD service accounts authenticate against GCP (via workload identity federation) before promoting any pipeline.

## Rollback

Keep the Azure workspace, ADLS storage, and Entra ID federation live through hypercare. Rollback is DNS/endpoint redirection back to Azure — do not decommission source identities or storage until the GCP target passes business validation.

## Automation opportunity

- Drive GCS bucket creation and service account provisioning from Terraform using the [Terraform patterns](/templates/terraform-patterns) as a base module.
- Generate the path-rewrite mapping from the [asset inventory](/discovery/asset-inventory) output rather than hand-maintaining it.
